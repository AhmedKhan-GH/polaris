Polaris · Developer Handbook

Adding a feature & integrating its dev API

Every Polaris feature is a self-contained folder with one public import surface. This page walks the playbook end to end: copy the exemplar, build under TDD, and publish exactly what outsiders may import.

Companion to CONTRIBUTING.md — the tracked markdown (DOMAIN-CHARTER.md → HANDBOOK.md → CONTRIBUTING.md) remains the source of truth.

The mental model

The foundation (lib/**) never imports features. Features (app/_features/<name>/**) never import each other. The only foundation files a feature PR touches are the three composition roots in lib/registry/. These are Iron Rules — enforced by a verification test that walks the import graph and fails the build on violations, not by convention.

Each feature exposes two distinct seams, for two distinct consumers:

SeamFile(s)ConsumerMay contain
Dev API index.ts Route pages and layouts Components, server actions, types — the supported surface
Manifests schema.ts · permissions.ts · nav.ts lib/registry/* only Table slice, ability contributor, nav entry — wiring, never logic

The two never mix: the index must not re-export manifests, and registries import manifests by name and nothing else. Anything a feature does not export from index.ts is private — free to rename, split, or delete without consulting anyone.

Dev API means import surface. This is a build-time contract between developers, not a web API — no HTTP endpoints, no runtime change. Pages call server actions exported from the index; those actions self-guard (permission → rate limit → validation → user context).

The playbook

Nine steps, in order. Every behavior starts as a failing test; every commit is small, green, and self-contained. Every code fragment below is taken verbatim from the notes exemplar in this repo (trims marked with …) — what you write is the renamed twin of what you see.

  1. Branch off the trunk

    feature/<name>, Conventional Commits (feat(orders): …), rebase + fast-forward merge when green.

  2. Copy the exemplar

    cp -r app/_features/notes app/_features/<name> — then rename: table, CASL subject (singular, 'Order'), topics (<name>:{userId} / <name>:all), limiter key, policy names, trigger names — and the exports in index.ts, which is now your dev API.

  3. TDD, no exceptions

    Red for the right reason → minimal green → refactor. A test that passes before its implementation lands belongs to an earlier commit or is redundant. Unit tests mock every foundation seam and assert the contract — here, pipeline order:

  4. Schema slice + RLS

    The table lives in your schema.ts with RLS enabled so it travels with the schema:

    Policies and grants are hand-written in the migration (npm run db:generate, then append). Pick the correct identity path: app-path tables use GUC policies, Supabase-path tables use auth.uid() — never mix the two on one table:

  5. Register the manifests

    One line each in the three composition roots — the only foundation files you touch. Flat lists, zero logic; removing a feature is deleting its lines:

    The manifests themselves stay in your folder — e.g. the ability contributor:

  6. Actions self-guard

    The pipeline order is contractual: guard → limiter → validate → context, and revalidate only after the whole chain resolves. The proxy is route hygiene, never an authorization layer.

  7. Realtime via the templates

    Broadcast trigger + channel policy copied from lib/realtime/templates/ into a --custom migration. Delivery is gated at the channel: a subscriber reads their own topic, an owner additionally reads the firehose. Never row-RLS the streamed table for delivery filtering.

  8. Publish the dev API

    Curate index.ts to exactly what your route page needs — components and actions outsiders consume. Everything unexported stays private. Then wire the page through it:

  9. Test all three tiers, run the gates

    Unit (mocked collaborators), integration (real app_user through RLS), E2E journeys (seeded owner/member). Then the gates below — every commit.

If you need to change lib/** outside the registries: stop. That is a charter conversation requiring an ADR, not a feature PR.

Evolving a dev API

Publishing a new symbol

Adding an export line to your feature's index.ts is the act of publishing. The failing test that justifies it is the consumer's: a page test imports archiveOrder from '@/app/_features/orders' and fails; one export line makes it green. Feature-owned file — always a feature PR, never a charter conversation.

Keeping things private

Hooks, row mappers, limiter instances, anything only your own files use: leave it out of the index. The boundary test proves no outsider depends on it, so internal refactors carry zero blast radius.

Wanting another feature's API

Forbidden, index or not — features never import each other. If two domains genuinely need to talk, that is a charter conversation, like the single sanctioned shell → auth edge.

The gates

Run what exists, every commit:

npm run lint && npx tsc --noEmit && npm test   # always
npm run test:integration                         # Docker required
npm run test:e2e                                 # local `supabase start` required
SKIP_ENV_VALIDATION=1 npm run build              # when routes/components changed

Live suites self-skip without the local stack; CI runs them with CI_REQUIRE_LIVE_DB=1, where skipping is a hard failure.
